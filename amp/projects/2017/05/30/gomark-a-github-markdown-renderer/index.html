Intro

When building out documentation for projects on github, I’ve found that writing everything in a .md markdown file is both the simplest and most pleasant solution to solving the problem of documentation.

A problem then occurs, however, when I try and see the output file before committing. In the past, I could rely on grip, but a huge limitation was the inability to render markdown when offline. As the project development seems to have stalled (last commit was 3 months ago, with 4 outstanding PRs and 20 open issues), the need for a new markdown renderer project became apparent.

This markdown renderer project will render markdown files locally, even when offline, allowing you to see how they look before committing and pushing to Github.

More than that, it uses the latest Github GraphQL v4 API, enabling for more efficient calls than the v3 REST API used by grip.

Features

  • Server application that will re-render file automatically as it changes- will not need a page refresh either
  • Built in Golang, making it extremely fast
  • Allows for offline rendering based on prior cached responses
  • Allows for HTML file output generation

Installation

MacOS

brew install grender

Others

go get -u github.com/JasonYao/gRender/cmd/gRender

Usage

Rendering a Markdown file

grender SOME_MARKDOWN_FILE.md

Rendering All Markdown Files in a Directory

grender SOME_DIRECTORY_WITH_MARKDOWN_FILES

Generate HTML Output

Single File Output

grender SOME_MARKDOWN_FILE.md -o OUTPUTFILE.html

Directory Output

grender SOME_DIRECTORY_WITH_MARKDOWN_FILES -o OUTPUT_DIRECTORY

Seeing the Help Menu

grender -h

Seeing the Help Submenu of a Command

grender SOME_COMMAND -h

Equivalence With Github’s Online Renderer

The goal for this project is two-fold:

  • Match Github’s online rendering exactly, or close to exactly in offline situations
  • Provide offline rendering when there is no access to the internet or rate limits are hit.

In order for this to occur, gRender will make a call to github’s markdown API. If it fails for rate limiting or connectivity issues, it will automatically fall back to the offline renderer, with any cached values from the last call to Github overriding shipped defaults.

It should be noted that any calls to Github will be over HTTPS, encrypting your data when it’s transported over the wire.
